# Eidolon — Database Guide (MongoDB Atlas)

This document explains **how Eidolon connects to the database**, **where every
database-related file lives**, and **what each one does**. It's written so that
anyone on the team can understand the data layer without digging through code.

> **Database engine:** MongoDB (hosted on **MongoDB Atlas**, free tier)
> **Database name:** `gamehub`
> **ORM / access layer:** [Prisma](https://www.prisma.io/) (`provider = "mongodb"`)

---

## 1. The big picture (how a request reaches the database)

```
React client (port 5173)
        │  HTTP (axios)
        ▼
Express API server (port 4000)
        │
Controller  →  Service  →  Prisma Client  ──────►  MongoDB Atlas
(routes)      (logic)      (server/src/lib/prisma.ts)   (cloud, "gamehub" db)
```

- The **client never talks to the database directly.** It only calls the API.
- Every database read/write goes through the **Prisma Client**, which is
  configured once in [server/src/lib/prisma.ts](server/src/lib/prisma.ts).
- Prisma reads the **connection string** from the `DATABASE_URL` environment
  variable, which lives in [server/.env](server/.env).
- The **shape of the data** (collections, fields, relations) is defined in
  [server/prisma/schema.prisma](server/prisma/schema.prisma).

Everything database-related lives inside the **`server/`** folder. The client
has no database code at all.

---

## 2. Where the files are (locations + explanation)

All paths are relative to the project root (`web project(react)/`).

| File | Location | What it does |
|---|---|---|
| **Connection string** | [`server/.env`](server/.env) | Holds the secret `DATABASE_URL` that points Prisma at the Atlas cluster. **This is the only place the real credentials live.** Never commit or share it. |
| **Schema (data model)** | [`server/prisma/schema.prisma`](server/prisma/schema.prisma) | Defines every collection (User, Game, Order, etc.), their fields, types, and relationships. This is the **source of truth** for the database structure. |
| **Prisma Client setup** | [`server/src/lib/prisma.ts`](server/src/lib/prisma.ts) | Creates the single shared `prisma` object the whole app imports. Adds **auto-retry** logic for flaky Atlas free-tier connections. |
| **Env loader** | [`server/src/lib/env.ts`](server/src/lib/env.ts) | Loads `server/.env` and exposes `env.databaseUrl`. Throws on startup if `DATABASE_URL` is missing. |
| **Health check** | [`server/src/controllers/health.controller.ts`](server/src/controllers/health.controller.ts) | The `/api/health` endpoint pings the DB so you can confirm the connection is alive. |
| **Generated client** | `server/node_modules/@prisma/client` | Auto-generated by `prisma generate`. **Do not edit** — it's rebuilt from the schema. |

### Files that actually read/write the database

The data layer is split so that **services** contain the database logic and
**controllers** wire it to HTTP routes. These 19 files import the Prisma client:

**Services** (`server/src/services/`)
- `auth.service.ts` — users & login
- `games.service.ts` — game catalog queries
- `cart.service.ts` — shopping cart
- `checkout.service.ts` — creating orders
- `fulfillment.service.ts` — granting games after payment (library entries)
- `review.service.ts` — reviews & votes
- `rawgSync.service.ts` — importing games from the RAWG API
- `chatAgent.service.ts` — AI chat sessions/messages

**Controllers** (`server/src/controllers/`)
- `health.controller.ts`, `me.controller.ts`, `wishlist.controller.ts`,
  `subscription.controller.ts`, `chat.controller.ts`
- Admin: `admin/users.controller.ts`, `admin/games.controller.ts`,
  `admin/analytics.controller.ts`, `admin/upload.controller.ts`

**Scripts** (`server/src/scripts/`)
- `syncGames.ts` — bulk-import games into the DB (`npm run sync:games`)
- `backfillPrices.ts` — one-off data fix-up

## 3. The connection string explained

The real connection string is in [`server/.env`](server/.env) under
`DATABASE_URL`. **Credentials below are redacted** — get the actual values from
`server/.env`, never from this document.

```ini
DATABASE_URL="mongodb://<USER>:<PASSWORD>@ac-...-shard-00-00.vq2x5ic.mongodb.net:27017,...:27017,...:27017/gamehub?ssl=true&replicaSet=atlas-...-shard-0&authSource=admin&retryWrites=true&w=majority"
```

What each part means:

| Part | Meaning |
|---|---|
| `mongodb://` | Standard (non-SRV) MongoDB protocol. Used locally because some networks can't resolve the `mongodb+srv` DNS record. |
| `<USER>:<PASSWORD>` | The Atlas database user credentials. |
| `...shard-00-00 / 01 / 02 ...:27017` | The three replica-set members of the Atlas cluster. Listing all three directly is what makes the non-SRV form work without DNS. |
| `/gamehub` | **The database name.** All collections live here. |
| `ssl=true` | Encrypted connection (required by Atlas). |
| `replicaSet=atlas-...-shard-0` | Tells the driver these three hosts form one replica set. |
| `authSource=admin` | Where the user credentials are authenticated. |
| `retryWrites=true&w=majority` | Standard reliability settings — retry failed writes, confirm them on a majority of nodes. |

There is also a commented-out **SRV form** in `.env`
(`mongodb+srv://...cluster0.vq2x5ic.mongodb.net/gamehub`). It's shorter and
preferred for cloud deployment hosts (which resolve SRV fine), but the direct
form above is more reliable for local development.

> **Atlas free tier note:** connections occasionally drop momentarily. That's
> why [`prisma.ts`](server/src/lib/prisma.ts) retries transient failures up to
> 4 times with backoff, so a hiccup recovers silently instead of erroring.

## 4. The collections (data model)

Defined in [`server/prisma/schema.prisma`](server/prisma/schema.prisma). Each
Prisma `model` becomes a **collection** in the `gamehub` database:

| Collection (model) | Purpose |
|---|---|
| `User` | Accounts: email, password hash, display name, role, Stripe customer id. |
| `Game` | The game catalog: title, description, media, price, ratings, etc. |
| `Order` | A purchase or subscription record (links a User to a Game). |
| `LibraryEntry` | A game a user owns / has access to (their "library"). |
| `Review` | A user's review of a game (rating, title, body). |
| `ReviewVote` | "Helpful" votes on a review. |
| `WishlistItem` | A game saved to a user's wishlist. |
| `CartItem` | A game in a user's shopping cart. |
| `ChatSession` | An AI assistant conversation owned by a user. |
| `ChatMessage` | An individual message within a chat session. |

Relationships (e.g. a `User` has many `Order`s; an `Order` belongs to one
`User` and one `Game`) are declared in the same schema file via Prisma
`@relation` fields.

## 5. Common database commands

Run these from inside the **`server/`** folder:

```bash
# Regenerate the Prisma client after editing schema.prisma
npm run prisma:generate

# Push the schema to the Atlas database (creates/updates collections & indexes)
npm run prisma:push

# Import games into the database from the RAWG API
npm run sync:games
```

To **view the data visually**, you can either:
- Open **MongoDB Atlas → Browse Collections** in the web dashboard, or
- Run `npx prisma studio` from `server/` (opens a local DB browser in your
  browser).

## 6. Connecting from another machine (for a teammate)

1. Make sure your computer's IP is allowed in **Atlas → Network Access**
   (for development, `0.0.0.0/0` allows everywhere — fine for testing, not for
   production).
2. Confirm the Atlas **cluster is not paused** (free clusters auto-pause after
   inactivity — resume it from the dashboard).
3. Put the real `DATABASE_URL` into your own `server/.env` (copy it from a
   teammate securely — **not** through this document, chat, or git).
4. From `server/`, run `npm run prisma:generate`, then `npm run dev`.
5. Verify the connection: open <http://localhost:4000/api/health> — you should
   see `"database":"ok"`.

## ⚠️ Security reminder

[`server/.env`](server/.env) contains the **live database password** and the
**Stripe secret key**. Anyone who gets that file has full read/write access to
the database and can move money in the Stripe account.

- **Never** commit `.env` to git or paste it into chat/docs/screenshots.
- Share credentials only through a secure channel (a password manager, etc.).
- If the connection string ever leaks, rotate the password in
  **Atlas → Database Access** immediately.
